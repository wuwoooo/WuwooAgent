# 当前结论与建议

## 一、当前结论

经过这几轮排查，当前可以比较明确地得出下面几个结论：

1. `Agent IME` 的文本注入与自动发送链路是可用的。
   - 能自动聚焦输入框
   - 能通过 IME 注入文本
   - 能自动点击发送

2. 当前问题不在 OCR，也不在图片保存链路。
   - 聊天区裁剪图和增强图都识别不到文本
   - 后续新增的 `raw` 原始帧直存图也同样模糊
   - 说明不是裁剪、增强、PNG 保存导致的模糊

3. 当前问题也不是手机整体显示设置导致的。
   - 同一页微信聊天窗口，系统手动截图是清晰的
   - 但 `MediaProjection` 截出来的主图与 raw 图都模糊

4. 因此，问题已经收敛到：
   - **这台手机上，微信页面通过 `MediaProjection` 输出到 `VirtualDisplay / ImageReader` 时，源帧本身就被模糊化了。**

## 二、已经证伪的方向

下面这些方向已经投入过排查，当前看都不是主要根因：

1. OCR 参数或 OCR 引擎本身
2. 聊天区裁剪范围
3. 图片增强参数
4. `Bitmap -> PNG` 保存过程
5. 多帧采样节奏
6. 截图前等待时长
7. 是否先聚焦输入框再截图
8. 是否重复拉起微信
9. `ImageReader -> Bitmap` 的尺寸拷贝链路

## 三、当前保留的有效策略

当前项目里已经保留了几个有效且必要的策略：

1. 当截图质量不达标且 OCR 为空时，**停止自动发送**
   - 避免糊图继续误触发回复

2. 保留日志与导出图片
   - 主图
   - raw 原始帧图
   - 聊天区裁剪图
   - 增强裁剪图

3. 主链路继续以 `agent-ime` 为核心
   - 输入
   - 注入
   - 发送

## 四、最重要的技术判断

当前最重要的判断是：

**继续在 `MediaProjection` 这条链路上做微调，预计收益已经很低。**

原因：

1. 系统手动截图清晰，证明屏幕本身内容没问题
2. `MediaProjection` 主图与 raw 图都模糊，证明模糊发生在更上游
3. 多轮策略调整后，截图模糊状态没有本质改善

## 五、后续建议

### 方案 A：接受当前结论，停止继续深挖 `MediaProjection`

适合场景：
- 以尽快验证整体闭环为目标
- 不再投入更多时间在当前机型的截图兼容性上

建议动作：
1. 保留当前失败保护
2. 不再继续投入 `MediaProjection` 优化
3. 后续换截图来源

### 方案 B：更换截图来源

这是当前最值得投入的技术方向。

可选路线：
1. `Shizuku + screencap`
2. `adb screencap`
3. `root`
4. 更换一台对 `MediaProjection` 兼容更好的测试机

### 方案 C：更换测试设备做快速验证

这是成本最低、最值得优先做的现实方案。

验证方式：
1. 换一台安卓机
2. 装当前版本 `agent-ime`
3. 同样授权截图并测试微信聊天页
4. 看 `MediaProjection` 截图是否仍然模糊

如果换机后截图清晰，就说明：
- 当前问题主要是 **这台手机 / 当前 ROM / 当前微信版本** 的组合兼容性问题

## 六、推荐决策

当前最推荐的决策顺序：

1. 保留现在的 `agent-ime` 主流程和失败保护
2. 不再继续深挖当前手机上的 `MediaProjection`
3. 优先做：
   - 换测试机验证
   - 或接更高权限的截图方案

## 七、一句话结论

**当前 `agent-ime` 的核心自动化链路已经跑通；剩余瓶颈不是 OCR，也不是代码后处理，而是当前设备上 `MediaProjection` 截微信时源帧本身被模糊化。**
